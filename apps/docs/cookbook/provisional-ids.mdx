---
title: Provisional Chat IDs
description: Preserve first-message state across `/` to `/chat/:id` navigation with draft IDs and bootstrap entries
---

This pattern keeps the first message and request metadata intact while the URL moves from `/` to `/chat/:id`.

## The Problem

The first-message flow has two competing requirements:

1. The user starts on `/`
2. The first message should persist under `/chat/:id`
3. The submitted message must survive the page transition
4. Switching to another chat still must remount and reset state

If the writable chat UI is mounted separately by both pages, Next.js route navigation remounts the runtime. If you bypass Next with `window.history.pushState`, the URL changes but the App Router tree does not.

## The Solution

Use a provisional client-generated chat id for draft routes and store a `ChatBootstrapEntry` before navigating to the persisted route.

- `/` and `/project/:projectId` use draft ids generated on the client by `useDraftChatId`
- the first persisted chat reuses that same id
- the submitted message and parallel request specs are stored in a bootstrap entry
- the app navigates with `router.push("/chat/:id")`
- `ChatPage` and `ProjectChatPage` read the bootstrap entry and pass it into `ChatSystem`

When the user switches to a different chat or starts a new one, the key changes and the runtime remounts on purpose.

## Architecture

### 1. Draft Routes Own Draft Identity

`ChatHome` and `ProjectPage` ask for a draft id when they render writable routes that do not yet have a persisted chat id:

```typescript
const id = useDraftChatId(projectId);
```

The hook keeps separate home and project draft versions. `resetDraftChatId` advances the version after a provisional chat is promoted so a later fresh draft gets a new id.

### 2. Pages Mount The Writable Runtime

The pages mount `ChatSystem` directly:

```tsx
<ChatSystem
  id={id}
  initialMessages={[]}
  persistedQueriesEnabled={false}
  routeSource="home"
/>
```

Persisted routes use `ChatRouteSystem`, which loads persisted messages unless a bootstrap entry is available.

### 3. Bootstrap Entries Bridge The Navigation

Before navigating, `useStartProvisionalChat` stores the submitted message and request specs:

```typescript
setChatBootstrap(
  createChatBootstrapEntry({ chatId, message, projectId, requestSpecs })
);
```

`ChatPage` and `ProjectChatPage` call `useResolvedChatBootstrap(chatId)` and pass the result to `ChatSystem`. `ChatSync` sends the primary request, creates pending assistant placeholders, and starts secondary parallel requests after the server confirms the chat.

### 4. Navigation Uses The App Router

For the first-message flow, the input navigates with the real router:

```typescript
router.push(projectId ? `/project/${projectId}/chat/${chatId}` : `/chat/${chatId}`);
```

This keeps the App Router, browser history, and chat route state in sync.

### 5. Full Resets Still Happen Via Key Changes

The remount strategy is preserved:

- `/` draft -> `/chat/:id` with the same id: bootstrap replays the submitted message
- `/chat/a` -> `/chat/b`: key changes, full remount
- `/chat/a` -> `/`: new provisional id, full remount
- tree branch switch inside one chat: same chat id, `threadEpoch` changes, `ChatSync` remounts

## Flow

1. Visit `/` -> `ChatHome` calls `useDraftChatId()` and renders `ChatSystem` with draft id `abc123`
2. Type and send -> `buildDraftChatSubmission` creates the user message and request specs
3. `useStartProvisionalChat` stores a `ChatBootstrapEntry` for `abc123`
4. Client navigates with `router.push("/chat/abc123")`
5. `ChatPage` calls `useResolvedChatBootstrap("abc123")`
6. `ChatSystem` receives the bootstrap entry and `ChatSync` sends the primary request
7. When bootstrap settles, the entry is cleared and persisted message queries resume
8. Later navigation to another chat or a fresh home draft changes the key and resets state

## Benefits

- No flicker on the first-message transition
- Real Next.js navigation instead of synthetic history mutation
- Explicit runtime resets when switching chats
- Preserved branch-reset behavior via `threadEpoch`
