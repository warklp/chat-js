---
title: Provisional Chat IDs
description: Preserve chat runtime state across `/` to `/chat/:id` navigation with a layout-owned shell
---

This pattern keeps the writable chat runtime mounted while the URL moves from `/` to `/chat/:id`.

## The Problem

The first-message flow has two competing requirements:

1. The user starts on `/`
2. The first message should persist under `/chat/:id`
3. The `useChat` runtime must not remount during that transition
4. Switching to another chat still must remount and reset state

If the writable chat UI is mounted separately by both pages, Next.js route navigation remounts the runtime. If you bypass Next with `window.history.pushState`, the URL changes but the App Router tree does not.

## The Solution

Use a provisional client-generated chat id, but mount the writable chat runtime once in the shared `(chat)` layout.

- `/` uses a draft id generated on the client
- the first persisted chat reuses that same id
- the app navigates with `router.push("/chat/:id")`
- the layout-owned runtime keeps the same key, so `useChat` stays mounted

When the user switches to a different chat or starts a new one, the key changes and the runtime remounts on purpose.

## Architecture

### 1. `ChatIdProvider` Owns Draft Identity

The provider keeps a provisional id for writable routes that do not yet have a persisted chat id:

```typescript
const [{ provisionalChatId, confirmedChatId }, setChatIdState] = useState(() => ({
  provisionalChatId: generateUUID(),
  confirmedChatId: null,
}));
```

For `/chat/:id`, the route id wins. For `/`, the provider exposes the current provisional id.

### 2. The Writable Chat Runtime Lives In Layout

The shared chat layout mounts a persistent client shell:

```tsx
<KeyboardShortcuts />
<PersistentChatShell />
{children}
```

That shell renders the writable `ChatSystem` for `/` and `/chat/:id`. The pages themselves become thin route/data wrappers.

### 3. The Runtime Key Stays Stable Across First Send

`ChatSystem` still uses the chat id as the remount key:

```tsx
<DataStreamProvider key={id}>
  <CustomStoreProvider<ChatMessage> initialMessages={initialMessages} key={id}>
```

On `/`, `id` is the provisional id. On the first send, the client persists the chat under that same id, then navigates to `/chat/:id`. Because the key does not change, the runtime does not remount.

### 4. Navigation Uses The App Router

For the non-project first-message flow, the input navigates with the real router:

```typescript
if (pathname === "/") {
  router.push(`/chat/${chatId}`);
}
```

This keeps the App Router, browser history, and chat runtime in sync.

### 5. Full Resets Still Happen Via Key Changes

The remount strategy is preserved:

- `/` draft -> `/chat/:id` with the same id: no remount
- `/chat/a` -> `/chat/b`: key changes, full remount
- `/chat/a` -> `/`: new provisional id, full remount
- tree branch switch inside one chat: same chat id, `threadEpoch` changes, `ChatSync` remounts

## Flow

1. Visit `/` -> provider creates provisional id `abc123`
2. Type and send -> local runtime still uses `abc123`
3. Client persists under `abc123`
4. Client navigates with `router.push("/chat/abc123")`
5. Layout shell stays mounted
6. `ChatSystem` key stays `abc123`, so `useChat` does not remount
7. Later navigation to another chat or a fresh home draft changes the key and resets state

## Benefits

- No flicker on the first-message transition
- Real Next.js navigation instead of synthetic history mutation
- Explicit runtime resets when switching chats
- Preserved branch-reset behavior via `threadEpoch`
